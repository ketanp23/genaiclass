0.1
Foundations definitions

Agent vs. agentic system

Before the cases, two terms worth pinning down — because they are used loosely and the difference matters for how you build, evaluate and govern. An AI agent is a single goal-directed worker. An agentic AI system is the whole engineered solution those workers live in. One is a component; the other is the product.

1 · What is an AI agent?

Definition · AI agent

An AI agent is a single unit, built around a language model, that pursues a goal over multiple steps rather than producing a one-shot answer. It takes in context, reasons and plans, uses tools to act on the world, observes what happened, and repeats — looping until the goal is met or it hands off. The defining move is that it acts, not just responds.

That makes an agent different from a plain chatbot or a single model call. A chatbot answers a question; an agent can look something up, decide what to do, do it, check the result, and try again. Four ingredients make it work, plus the loop that ties them together.

Examples of single AI agents

  • A coding agent told "make this failing test pass" — it reads the repository, edits files, runs the test, sees the error, and keeps revising until it is green.
  • A customer-service agent that looks up an order through an API, checks the returns policy, and issues a refund within an allowed limit.
  • A research agent that searches the web, opens and reads pages, and assembles a cited summary on a topic.
  • A scheduling agent that reads a calendar, finds workable times, and sends the invitations.
Tie-in

Every worker in the cases ahead is an individual AI agent — the Transaction analyser in the bank, the Pricing & promo agent in the retailer, each simulation agent in the manufacturer. On its own, each is just one agent doing one job.

Where "agent" sits among the words it's confused with

It helps to place agent next to its neighbours. A widely-shared 2×2 sorts AI systems by two dials: how much it runs without you (automation, low→high, left→right) and how deeply the system is woven into your tools and workflows (integration, low→high, bottom→top).

Read the four corners as a progression. Assistants respond when prompted and leave you in control. Copilots work alongside you inside your tools, proposing as you go, but you still accept or reject. Autopilots run a fixed routine on their own without reasoning about it. Agents sit top-right — highly automated and deeply integrated — pursuing a goal over multiple steps and acting through your systems to reach it. That corner is what this playbook is about. The boundaries blur in practice, and products migrate toward it as they take on more automation and tie into more systems, so treat the placement as a way to think, not a hard taxonomy.

Levels of autonomy

Autonomy isn't all-or-nothing. It runs along a spectrum — and within any single use case it is a dial you set deliberately, not a fixed property of "the AI". At one end the model only answers; at the other, systems of agents run a whole outcome.

More granularly, it pays to borrow how the automotive industry grades self-driving and talk about a system's autonomy in levels. The level is a choice you make per use case — matched to how reversible and consequential the actions are.

LevelPostureHuman roleWhere it fits
L0ManualHuman does everything; AI not involvedHighest-stakes, novel judgement
L1AssistedAI drafts; human edits & executesDocument drafting, summarisation
L2Human-in-the-loopAgent proposes an action; human approves before it executesCustomer refunds, code merges, payments
L3Human-on-the-loopAgent acts; human monitors & can intervene/overrideTriage, routing, monitoring at scale
L4Conditional autonomyAgent acts independently within a defined envelope; escalates edge casesReconciliation, routine procurement
L5Full autonomyAgent owns the outcome; humans review in aggregateRare today; only fully reversible, low-stakes work
Framework · the autonomy / consequence rule

Grant the lowest level of autonomy that still captures the value, and raise it only as evidence of reliability accumulates. The right starting level turns on two questions: how reversible is the action? and how costly is a wrong one? Reversible and cheap → start higher (L3–L4). Irreversible or expensive → start lower (L1–L2), with a human gate, regardless of how confident the demo looked.

Facilitator note · the highest-leverage idea

If an audience remembers one thing, make it this: autonomy is a dial you set per use case, not a property of "the AI." Most failed agentic pilots set the dial too high too early — granting L4 autonomy to a process whose mistakes were expensive and hard to reverse. Every case in this playbook picks its level deliberately.

Anatomy of an AI agent

Strip away the marketing and every agent is the same loop wrapped around five components. Knowing the parts lets you see where an agent will fail — and where the money goes.

The agent loop

We sketched this loop earlier; here it is in full, including the reflection step that separates robust agents from brittle ones. An agent is not a single call to a model — it is a loop: the model thinks, chooses an action, the action runs in the real world, the result comes back, and the model reconsiders. The loop repeats until the goal is met or a stop condition fires.

Two practical consequences follow. First, agents are loops, so they consume model calls repeatedly — cost scales with the number of steps, not the number of requests. Second, every pass through "Act" touches a real system; that is precisely where permissions, logging and guardrails must live.

The five components

1Model

The reasoning engine that plans and decides. The choice trades off capability, latency and cost — many systems use a strong model for planning and cheaper ones for routine sub-steps.

2Memory

Short-term (the working context of the current task) and long-term (facts, past interactions and knowledge retrieved on demand). Poor memory is the most common cause of agents "forgetting" or repeating themselves.

3Tools

The agent's hands: APIs, databases, search, code execution, other software. An agent is only as capable as the tools it can call — and only as safe as those tools' permissions.

4Planning

How the agent decomposes a goal into steps and sequences them. Ranges from simple step-by-step reasoning to explicit plans it can revise mid-task.

5Orchestration

The control logic that runs the loop: when to stop, when to retry, when to escalate to a human, how to handle errors. This is where reliability is engineered.

Where quality actually comes from

Leaders often assume agent quality is mostly about the model. In knowledge-intensive use cases it usually isn't. Industry benchmarking through 2026 attributes the majority of an agent's real-world performance to the quality of the context it retrieves — the data, documents and memory feeding the model — rather than the model itself. Your proprietary data and how you wire it in is the moat; the model is increasingly a commodity.

Agentic design patterns

A handful of reusable patterns underpin almost every production system. You don't need to implement them, but recognising them helps you read a proposal critically.

Pattern · reasoning
Reason-and-act (ReAct)

The agent alternates between thinking and acting, using each observation to inform the next thought. The default backbone of single agents.

Pattern · quality
Reflection

The agent critiques its own output and revises it — or a second "critic" agent reviews the first. A cheap, high-impact way to raise reliability.

Pattern · structure
Planning & decomposition

The agent first writes an explicit plan, then executes the steps. Useful for long, multi-stage tasks where drift is a risk.

Pattern · scale
Orchestrator–workers

A lead agent breaks the goal into sub-tasks, dispatches them to specialised workers, then synthesises the results. The core of agentic AI.

Framework · workflows vs. agents

A critical distinction from current engineering practice: a workflow follows a predefined path (the steps are fixed in code; the model fills in the blanks), whereas an agent dynamically decides its own steps. Workflows are more predictable, cheaper and easier to govern; agents are more flexible but less deterministic. The mature default is to use the simplest thing that works — start with a structured workflow and add open-ended agency only where the problem genuinely demands it. Many "agentic" projects fail because they reached for full autonomy when a deterministic workflow would have been safer and cheaper.

2 · What is an agentic AI system?

Definition · agentic AI system

An agentic AI system is the complete architecture in which one or more agents do real work safely and reliably. It is the agents plus everything around them: an orchestration that coordinates them, grounded knowledge they retrieve from, tools reached over governed protocols, the human gates and guardrails on consequential action, and the observability, identity and evaluation that make it trustworthy. The agent is the engine; the agentic system is the whole vehicle.

A single capable agent rarely solves a business problem by itself. Production work needs several specialists coordinated, anchored in the organisation's real data, allowed to touch real systems only within limits, watched, and owned by a named human. That engineered whole is the agentic system.

Examples of agentic AI systems

  • The AML investigation system in Case 01 — a lead orchestrator, four worker agents, retrieval over policy and prior cases, scoped read-only tools and a human filing gate — is an agentic system.
  • The retail merchandising fleet in Case 03 — a supervisor plus operational agents acting inside a deterministic guardrail envelope, with a kill switch — is an agentic system.
  • A software-engineering system that pairs a planner, a coder, a reviewer and a test-runner agent over a codebase, behind code review.
  • A supply-chain control tower that senses, simulates and recommends, with a planner committing the action — Case 04.

Holding the two apart

Why the distinction matters

Buying or building a clever agent is the easy 20%. The value — and nearly all the risk — lives in the surrounding system: the grounding, the permissions, the human gate, the evaluation and the audit trail. That is exactly why the rest of this playbook examines whole systems, not agents in isolation.

0.2
Orientation read first

How to read each case

Every case in this playbook is built the same way, so you can compare them directly. Each one moves through four lenses — the organisation it lives in, the agentic architecture that delivers it, how the system is evaluated, and the risks that must be controlled. The technology is broadly the same across all four; what changes is how context, consequence and regulation reshape it.

A
The organisation

Size, ownership, regulatory exposure and operating model. The profile determines how much autonomy is tolerable and where value sits.

B
The architecture

The agents, tools, data and orchestration that do the work — plus the human gates and governance rails wrapped around them.

C
The evaluation

How you know it works and keeps working: offline test suites, online quality sampling, business outcomes and guardrail metrics.

D
The risk posture

What can go wrong, how badly, and the concrete mitigation for each — the controls that earn the right to let the system act.

Framework · the autonomy / consequence rule

One idea runs through all four cases. The level of autonomy a business grants an agent should be inversely proportional to the cost of being wrong. Where an error is cheap and reversible, agents can act freely. Where an error is expensive, regulated, or irreversible — a wrongful denial of care, a mis-filed regulatory report, a bad commitment to a customer — a human gate sits between the agent's recommendation and the real-world action. Read each architecture diagram by first finding that gate.

01
Case one Financial services

The financial-crime investigator

Financial services · Global Tier-1 bank

Anti-money-laundering alert investigation

An agentic system triages and investigates transaction-monitoring alerts end to end — gathering evidence, resolving entities, screening for sanctions and adverse media, and drafting a defensible case narrative — so human investigators spend their time deciding, not assembling.

A · The organisation

Profile
Global systemically-important bank; hundreds of thousands of staff; operations across dozens of jurisdictions.
Pressure
Tens of thousands of monitoring alerts per month; ~90–95% are false positives; investigator backlogs and turnover are chronic.
Regulatory exposure
Extreme. BSA/AML, sanctions regimes, and model-risk supervision; multi-hundred-million-dollar fines for systemic failures.
Why agentic
Investigation is multi-step evidence work across many systems — a natural fit for a planning agent that orchestrates retrieval and reasoning, with a human owning the filing decision.

B · The agentic architecture

The pattern is orchestrator–workers. A lead agent decomposes each alert into an investigation plan and dispatches specialist worker agents in parallel; it then synthesises their findings into a single draft case. Crucially, the system recommends but never files — every Suspicious Activity Report passes through a human investigator.

Lead · Investigation orchestrator Transaction analyser Entity resolution Sanctions & adverse-media Narrative drafter
Figure 1 · Orchestrator–workers investigation architecture

Two cross-cutting rails wrap the whole flow. Observability records every prompt, tool call and intermediate finding to an immutable audit trail — essential when a regulator later asks why a report was or was not filed. Identity & least-privilege means each tool is reached with read-only, scoped credentials; no agent can move money or alter a customer record.

C · Evaluating the system

Because a missed case is a regulatory failure and a false case wastes scarce investigators, evaluation is taken as seriously as the model itself. It runs on three horizons: an offline test suite that gates every change, continuous online quality sampling, and the business outcomes that justify the programme.

Definition · trajectory vs. outcome evaluation

Outcome evaluation asks whether the final answer was right. Trajectory evaluation asks whether the agent got there sensibly — the right tools, in a reasonable order, without wasteful or risky detours. Agentic systems need both: a correct conclusion reached by a broken path will eventually fail on a case where the path matters.

Why this shape

In a Tier-1 bank the cost of being wrong is measured in regulatory enforcement, so autonomy is deliberately low: the agents do the labour of investigation and a human keeps the decision. The return is not headcount removal but throughput and consistency — more alerts cleared, with a defensible, audited rationale behind every one.

Case two Healthcare

The prior-authorisation coordinator

Healthcare · Regional non-profit health system

Prior authorisation & care coordination

An agentic pipeline reads incoming authorisation requests, checks them against payer medical policy and clinical guidelines, confirms eligibility, and clears the clearly-approvable cases automatically — routing every denial or doubtful case to a licensed clinician. Staff stop chasing paperwork and start coordinating care.

A · The organisation

Profile
Regional non-profit health system; a handful of hospitals and clinics; mission-driven, margin-thin, perennially short of administrative staff.
Pressure
Prior authorisation is slow, manual and a leading cause of care delays and clinician burnout; payer rules differ and change constantly.
Regulatory exposure
High and patient-facing. HIPAA, clinical-safety duties, payer contracts, and emerging rules on automated coverage decisions.
Why agentic
The work is a sequence of reference-and-check steps against documents — ideal for a grounded pipeline — but the duty of care means a human must own any decision that withholds treatment.

B · The agentic architecture

The pattern is a sequential pipeline with a human-in-the-loop gate. Each stage hands a structured result to the next. The defining design choice is an autonomy asymmetry: because granting care is low-harm and withholding it is high-harm, the system may approve a clearly-eligible request on its own, but it may never deny — denials and ambiguous cases always go to a clinician.

C · Evaluating the system

Evaluation centres on two things a health system cannot get wrong: clinical safety (never auto-approving something inappropriate, never wrongly steering toward denial) and equity (the system must not disadvantage any patient group). Speed matters, but only after safety is proven.

Framework · evaluate the abstention, not just the answer

In high-consequence settings the most important behaviour is often knowing when to stop. A system that confidently handles 100% of cases is more dangerous than one that handles 80% and reliably escalates the rest. So a core metric is the quality of the escalation decision itself — does the agent route the genuinely hard cases to humans?

Why this shape

The asymmetry is the whole design. Letting agents say yes removes the administrative drag that delays care; reserving every no for a clinician keeps the duty of care — and its accountability — firmly with a human. Value shows up as faster approvals and reclaimed staff time, not as removing humans from the decision that matters most.

Case three Retail

The merchandising & service operator

Retail · PE-backed mid-market omnichannel retailer

Merchandising operations & customer service

A supervisor agent coordinates a team of operational agents that replenish stock, tune pricing and promotions, maintain the catalogue, and answer customers — all acting on their own, but only ever inside hard, deterministic guardrails. Anything that exceeds those limits stops for a human.

A · The organisation

Profile
Private-equity-backed mid-market retailer; selling across web, marketplaces and stores; lean team under an explicit efficiency-and-growth mandate.
Pressure
Thousands of SKUs, constant price moves and a flood of routine customer queries — far more operational decisions than the team can make by hand.
Regulatory exposure
Moderate. Consumer protection, advertising and pricing rules, payment and data-privacy obligations — but errors are largely reversible.
Why agentic
High volume, low individual stakes, fast feedback — the profile where genuine autonomy pays off, provided it is fenced by firm operating limits.

B · The agentic architecture

The pattern is hierarchical with mixed autonomy. A supervisor agent routes work to specialist operational agents that act autonomously inside a guardrail envelope — deterministic, code-enforced limits such as margin floors, reorder budgets and refund caps. The model proposes; the envelope decides what is allowed to happen. Anything outside the envelope routes to a human, and an always-available kill switch can pause the fleet instantly.

Definition · the guardrail envelope

An envelope is a set of deterministic, non-AI limits enforced in code around the agents: maximum reorder spend, minimum margin, refund ceilings, approved-supplier lists, brand rules. The probabilistic model can be creative inside the envelope but physically cannot act outside it. This "non-deterministic core, deterministic boundary" pattern is what makes higher autonomy safe.

Why this shape

This is the inverse of the bank. Cheap, reversible, high-volume decisions justify real autonomy — so the agents act and humans supervise the exceptions. The safety does not come from a person checking each action; it comes from the deterministic envelope and the kill switch. Value is direct: margin, availability and service capacity that scale without scaling headcount.

Case four Manufacturing

The supply-chain disruption responder

Manufacturing · Multinational industrial conglomerate

Supply-chain disruption response

Always-on monitoring agents watch for disruption across suppliers, logistics, weather and demand. When something credible breaks, a response orchestrator spins up simulation agents to model mitigation options against a digital twin, ranks them by cost and impact, and hands a planner a decision-ready brief — but the commitment stays human.

A · The organisation

Profile
Multinational industrial conglomerate; multiple business units with their own P&Ls; decentralised, with a shared technology platform.
Pressure
Sprawling multi-tier supplier networks; disruptions (port closures, supplier failures, weather) detected late and resolved manually under time pressure.
Regulatory exposure
Moderate but varied — trade, safety and OT/IT-security obligations differ by unit and geography; commitments move real money.
Why agentic
Continuous sensing across many feeds plus rapid option-modelling is beyond manual reach; a planner can then choose well from agent-prepared scenarios.

B · The agentic architecture

The pattern is autonomous monitoring feeding human-gated simulation. Monitoring agents run continuously — sensing is cheap and reversible, so it is fully autonomous. On a credible signal, an orchestrator launches parallel simulation agents to model mitigations against a digital twin, then ranks them. Because committing a mitigation spends money and reallocates capacity, a planner commits the chosen plan; only then do execution systems act.

C · Evaluating the system

Evaluation splits cleanly along the autonomy line. The monitoring half is judged on detection quality; the simulation half on whether its recommended plans hold up against reality and against what expert planners would have chosen.

Framework · evaluate sensing and acting separately

When a system has an autonomous half and a human-gated half, evaluate them as two systems. Monitoring lives or dies on the recall/precision trade-off — miss too little, cry wolf too often. Simulation is judged on decision quality — would a good planner have chosen what the agents surfaced, and did reality bear out the modelled costs? Blurring the two hides where the system is actually weak.

Why this shape

Speed of insight is the prize; speed of commitment is the danger. The conglomerate gains earlier warning and far richer choices under pressure, without handing irreversible spend to a model.

Read across Synthesis

What the four cases share

Place the four cases side by side and the same skeleton appears every time. What differs is one dial — how much the business lets the agents do on their own — and that dial is set almost entirely by the cost of being wrong.

The autonomy spectrum

The same technology sits at very different points on the autonomy spectrum depending on context. Cheap, reversible, high-volume work earns real autonomy; costly, regulated, irreversible decisions keep a human in the loop.

One architectural skeleton

Strip away the domain detail and every case is built from the same six parts. Learn this shape once and you can read any agentic architecture put in front of you.

Part 1
An orchestrator

A planning agent that decomposes the goal and routes work — the lead investigator, the ops supervisor, the response coordinator.

Part 2
Specialist workers

Focused agents that each do one thing well, run in parallel or in sequence, and report back to the orchestrator.

Part 3
Grounded knowledge

Retrieval over governed sources with provenance — policy, guidelines, contracts — so claims are cited, not invented.

Part 4
Tools over protocols

Scoped, least-privilege access to real systems, increasingly via open protocols, so agents can read and (carefully) act.

Part 5
A human gate or envelope

The control on consequential action — a person who commits, or a deterministic limit the model cannot exceed.

Part 6
Observability & governance

Immutable audit, identity, evals and a kill switch wrapped around everything — the rails that make trust defensible.

The universal risk-and-mitigation map

The headline risks are the same in every case; what changes is their severity and how hard the controls are turned up.

Recurring riskThe recurring mitigation
Wrong or fabricated outputRetrieval grounding with citations; offline eval gates; trajectory checks
Prompt injection / poisoned dataTreat all external content as untrusted data, never instructions; sanitise inputs; keep tools scoped so a hijack cannot act
Over-action / runaway behaviourHuman gate at the point of consequence, or a deterministic envelope; rate limits; a kill switch
Privacy & data leakageLeast-privilege scoped access; data minimisation; no training on sensitive data; full audit
Bias & unfair outcomesFairness testing pre-launch and in production where people are affected; remediation triggers

Automation bias / over-relianceSurface uncertainty; track override rates; keep humans genuinely deciding, not rubber-stamping
The technology is general; the architecture is specific. You design an agentic system not around what the model can do, but around what it costs to be wrong.
Framework · five questions to size any agentic use case
  1. What does it cost to be wrong? Cheap and reversible invites autonomy; costly and irreversible demands a human gate.
  2. Where is the gate or envelope? Name the exact point where a human commits, or the deterministic limit the model cannot cross.
  3. What is it grounded in? Identify the governed source of truth and confirm every claim cites it.
  4. How will we know it works — and keeps working? Define the offline suite, online sampling and the one business metric that justifies it.
  5. What is the blast radius if it fails? Scope tool permissions, add a kill switch, and make sure the worst case is survivable.
Facilitator note · using these four cases

Run the cases in order of increasing autonomy comfort or, more powerfully, contrast case 01 (bank) with case 03 (retail) directly — same underlying technology, opposite autonomy postures — to make the consequence-driven design point land. Then ask the room to place one of their own candidate use cases on the autonomy spectrum and walk it through the five questions above. The goal is not to teach the diagrams but to teach the habit of locating the human gate first.

The substrate: two open protocols

Two standards now sit under most enterprise agentic systems, so teams stop hand-building integrations. Both were placed under open, foundation-led governance over the past year, which is what made them safe to standardise on.

Definition · MCP and A2A

Model Context Protocol (MCP) is the vertical standard — how an agent connects to tools and data. Created by Anthropic and donated to the Linux Foundation's Agentic AI Foundation in late 2025, it spawned thousands of reusable servers and is backed across the major model providers. Agent-to-Agent (A2A) is the horizontal standard — how agents discover and delegate to one another, including across vendors. Now a Linux Foundation project with adoption across the major clouds and 150+ organisations, it is what lets a multi-agent system span teams and business units. Put simply: MCP gives an agent hands; A2A gives it colleagues. Commercial transaction layers for agent-to-agent payments are emerging above these.

Choosing an orchestration approach

The topologies from the four cases are choices at the orchestration layer, and the frameworks differ in which they make natural. The consistent 2026 signal: graph-based, checkpointed runtimes are favoured wherever auditability and human-approval steps matter — exactly the regulated cases.

PatternSeen inWhat suits it

Sequential + human gateCase 02 · healthcareGraph runtimes with checkpoints and built-in human-in-the-loop approval — the regulated default
Hierarchical, mixed autonomyCase 03 · retailRole-based crews (CrewAI) or hierarchical agent trees (Google ADK) wrapped in deterministic guardrails
Monitor + simulateCase 04 · manufacturingEvent-driven runtimes with A2A coordination across decentralised business units

Framework character, broadly: LangGraph for stateful, auditable, human-gated workflows; CrewAI for fast multi-agent prototyping; OpenAI Agents SDK for low-friction hand-off agents; Google ADK for Gemini/GCP and A2A-native multi-agent trees; Microsoft Agent Framework (the unified successor to AutoGen and Semantic Kernel) for .NET and Azure estates; Anthropic Claude Agent SDK for MCP- and skills-native agents.

Run it as a platform, not a project

The decisive enterprise move is to stop building one-off agents and stand up a shared platform. A central team owns the control plane — model gateway, tool and MCP registry, identity, guardrails, evaluation and observability — while business-unit teams build agents on top within those rails. This is exactly how the decentralised conglomerate in Case 04 keeps consistency without central bottlenecks.

The identity control plane

The fastest-growing risk in 2026 is agent sprawl: autonomous identities created without owners, carrying excessive standing privilege. The emerging discipline treats every agent as a delegated non-human identity — inventoried, least-privilege by default, with just-in-time rather than standing access, and an auditable chain of delegation. Major platforms now ship agent-specific governance that unifies identity, data-protection and threat controls. The principle is blunt: an agent is a user with superpowers, so govern it like one.

Buy an agent and you have a demo. Build the platform — the rails, the protocols, the governance — and you have a capability.
On the currency of this stack

The layers, the two protocols and the two rails are durable; the specific products move every quarter. Analysts expect task-specific agents in a large share of enterprise applications by the end of 2026, and most enterprises are increasing investment — which is precisely why the market churns. Commit to the architecture; re-check the tooling before you build.

How teams choose an orchestration framework

The framework decision dominates a build, and there is now reasonable consensus on the trade-offs. You should expect your teams to justify their choice against these lines, and to favour open protocols over proprietary lock-in.

The pattern mature teams converge on

Most production systems do not bet on a single framework. The common shape is a controllable orchestration backbone (for state, checkpoints and human-in-the-loop gates) that delegates bounded sub-tasks to lighter agents inside it — and underneath, a deliberate investment in retrieval quality, since the data feeding the agents, not the framework, accounts for most of the real-world performance in knowledge-intensive work.

Reference Glossary

Glossary

Agentic AI
AI systems that pursue a goal over multiple steps — planning, using tools and acting — rather than answering a single prompt.
Orchestrator–workers
A topology where a lead agent decomposes a task and dispatches specialist worker agents, then synthesises their results.
Human-in-the-loop (HITL)
A design where a person must approve or make a specific decision before an agent's recommendation becomes a real-world action.
Guardrail envelope
Deterministic, code-enforced limits (spend caps, margin floors, refund ceilings) that bound what an autonomous agent is permitted to do.
Grounding / RAG
Retrieval-augmented generation: anchoring an agent's outputs in retrieved, governed source documents so claims can be cited.
Trajectory evaluation
Assessing the path an agent took — the tools and steps — not only whether the final answer was correct.
Abstention / escalation
An agent recognising the limits of its competence and routing a case to a human instead of acting.
Prompt injection
An attack where malicious instructions hidden in data an agent reads attempt to hijack its behaviour.
Least privilege
Granting an agent only the minimum, scoped access needed for its task — often read-only — to limit the blast radius.
Digital twin
A simulation model of a real system (e.g. a supply network) that agents can run what-if scenarios against.
LLM-as-judge
Using a language model, against a rubric, to score the quality of another model's outputs at scale, validated by human review.
Automation bias
The human tendency to over-trust an automated recommendation and approve it without genuine scrutiny.
Kill switch
A control that immediately pauses or halts an agent fleet — essential wherever agents act autonomously.
Blast radius
The maximum harm an agent could cause if it failed or were compromised — minimised by scoping and gates.
Orchestration
Coordinating multiple agents toward a goal — and, at enterprise scale, running the whole estate of agentic systems on shared rails.
MCP & A2A
Open protocols underneath agentic systems: MCP connects an agent to tools and data (vertical); A2A lets agents discover and delegate to each other (horizontal).
Control plane
The shared platform layer — model gateway, tool registry, identity, guardrails, evaluation, observability — that business teams build agents on top of.
Non-human identity
An agent treated as a governed identity: inventoried, least-privilege, with just-in-time access and an auditable chain of delegation.
