PHASE A Ingestion — offline, scheduled Building the knowledge base, before any user shows up
PHASE A · STEP 01
01
Pull from every source

Five different systems hold the knowledge an adjuster needs. The pipeline starts by collecting all of them into one place — a flat folder of raw content, untouched.

Diagram 1.1 — Source connectors → Unstructured Corpus
What the component does

Source connectors are plug-ins that authenticate to SharePoint, Outlook, Salesforce and read documents on a schedule. They preserve original filename, version, and date.

Why it matters

If a source is missed here, the model is permanently blind to it. Coverage of the input estate is the first determinant of answer quality.

PHASE A · STEP 02
02
Cut into clauses, tag each one

Long documents are useless to a model in one piece. The chunker breaks each PDF into individual clauses or sections, and stamps every chunk with metadata: policy variant, jurisdiction, issue date.

Diagram 1.2 — Chunking a policy document → Tagged chunks
Why section-aware chunking

A naive fixed-size cut would slice "§ 4.2 Outbuildings" in half. Section-aware chunking respects the clause boundary, so a clause is always retrieved whole.

Why metadata tagging

Tagging policy variant and date lets retrieval filter to policy:H-2019 only. Without tags, the model would see clauses from every variant — and pick the wrong one.

PHASE A · STEP 03
03
Turn meaning into vectors

Each clause is fed to an embedding model that converts its meaning into a long list of numbers. Clauses about similar concepts land near each other in this numerical space — even if they use different words.

Diagram 1.3 — Embedding & indexing → Vector store
What an embedding is, plainly

A vector of numbers that captures the meaning of a passage. Two clauses about wind damage land near each other even if one says "wind" and the other says "storm."

The vector store

A specialised database that, given a query vector, finds the closest matches across millions of records in milliseconds. This is what makes retrieval fast and meaning-aware.

PHASE B Query — live, per question What happens in the seconds after an adjuster asks
PHASE B · STEP 04
04
Adjuster asks in plain English

An adjuster types a real-world question into the co-pilot. The system attaches the claim's metadata — the policy variant, the customer's jurisdiction — so retrieval can filter to only the clauses that apply.

Diagram 1.4 — Question + context enters the pipeline → Query embedding
Why context matters

The same question on two claims has two different answers. Attaching policy, endorse, jurisdiction as retrieval filters prevents the system from mixing variants.

Same embedding model — both sides

The question is embedded by the same model that embedded the chunks in Step 3. Both live in the same numerical space, so "near each other" is meaningful.

PHASE B · STEP 05
05
Find the closest clauses, re-rank, filter

The vector store returns the 20 clauses whose meaning is nearest to the query. A re-ranker reads each one carefully and keeps the top 3. Permission and policy-variant filters trim anything that doesn't belong.

Diagram 1.5 — Retrieval & re-ranking → Top clauses
Two-stage retrieval, why?

Stage one (vector search) is fast but rough — it casts a wide net. Stage two (re-ranker) is slower but sharper — it inspects each candidate carefully. Combining them gets both speed and accuracy.

Filters fire here, not later

Permission, jurisdiction, and recency filters are applied during retrieval — never after the LLM has spoken. The model can never see what it shouldn't.

PHASE B · STEP 06
06
Generate the answer with citations

The top 3 clauses, the original question, and a strict instruction ("answer only from these passages, cite every claim") are handed to the LLM. The model produces a coverage opinion the adjuster can defend, line by line.

Diagram 1.6 — Augmented prompt → grounded answer → Cited response
The model is constrained, not free

The system prompt forbids invention. If the answer isn't in the three passages, the model must say so. This is the architectural cure for hallucination.

Every claim is clickable

Each citation links back to the original clause in SharePoint. A regulator, auditor, or disputing customer can trace any decision to its source — instantly.

USE CASE 1 · WRAP

The adjuster pressed one button. Behind it: six steps, five systems, 280,000 clauses, one cited answer in under three seconds. The model never invented anything — it only assembled what retrieval put on its desk.

PHASE A Schema preparation — offline Teaching the system what the business means by "revenue," "store," "active SKU"
PHASE A · STEP 01
01
Start with the warehouse, as it is

The retailer's Snowflake warehouse has hundreds of tables holding sales, returns, inventory, footfall and promo data. They refresh hourly. The pipeline does not duplicate this data — it learns its shape.

Diagram 2.1 — Warehouse landscape → Schema metadata
The schema problem

The warehouse has 14 columns that could plausibly mean "revenue." Picking the wrong one gives a confident, wrong, defensible-looking number. This is the hardest problem in structured-data RAG.

No data is copied

The pipeline reads the schema — table names, column names, sample values, relationships — but never duplicates the warehouse. Queries run live, so answers are always current.

PHASE A · STEP 02
02
Build the semantic layer

A small team — finance plus data — agrees on canonical business definitions. "Revenue" means one specific column with one specific filter. "Active store" means three specific conditions. The model is given these as gospel.

Diagram 2.2 — Business terms → canonical SQL → Semantic dictionary
The single most important step

The semantic layer is what separates a structured-RAG pilot from a structured-RAG product. Without it, every leader sees a slightly different version of "revenue."

Built once, used forever

About 120 business terms cover 95% of questions. Defining each one is a one-time investment that the model and every downstream tool reuses indefinitely.

PHASE A · STEP 03
03
Make the schema retrievable

Schema descriptions, sample queries, business term definitions, and historical successful SQL are all embedded into a vector store. When a question arrives, the system retrieves only the parts of the schema it needs — not all 4,800 columns.

Diagram 2.3 — Building the retrieval index for schema → Schema vector store
Why we still need retrieval

You can't paste 4,800 column descriptions into a prompt — too long, too noisy. Retrieval narrows the schema down to the ~10 columns relevant to the question being asked.

The past as a teacher

Including ~800 vetted question-to-SQL pairs lets the model imitate proven patterns rather than invent SQL from scratch. New questions reuse old solutions.

PHASE B Query — live, per question A store manager asks; the system writes SQL, runs it, narrates the answer
PHASE B · STEP 04
04
Store manager asks in plain English

A store manager asks a real question on her phone, the same way she'd ask a colleague. The system attaches her identity — her store, her permissions — so the answer is scoped to what she's allowed to see.

Diagram 2.4 — Question → relevant schema retrieved → Working context
Complex questions get split

"Why did I underperform?" actually contains three questions. Splitting them produces clean, debuggable SQL — and is easier for the model than one giant query.

Identity = permission

Priya's identity narrows scope to store_id = 14. She physically cannot see Bangalore's numbers — the filter is applied at retrieval, not by trusting the LLM.

PHASE B · STEP 05
05
Generate SQL, validate, execute

The LLM writes SQL using only the schema it was given. A validator checks the SQL is safe (read-only, scope-respecting, no joins to disallowed tables). It then runs against the warehouse — live, fresh, never copied.

Diagram 2.5 — SQL generation, validation, execution → Result rows
The validator is non-optional

The LLM cannot run a DELETE, an UPDATE, or a query that touches tables outside its allow-list. The validator rejects anything dangerous before it reaches the database.

The data never moves

The SQL runs against Snowflake live. No data copies, no stale extracts, no governance gap. The model sees rows but never sees the warehouse itself.

PHASE B · STEP 06
06
Turn rows into a narrative answer

The result rows go back into the LLM with one final instruction: "Explain these numbers to a store manager in plain English; show the SQL and the source tables." The output is a chat message a non-analyst can act on.

Diagram 2.6 — Rows → narrated answer with citations → Manager's screen
Narration with discipline

The LLM is told to use only the rows it was given. It can rank, summarise, recommend — but cannot fabricate a number. Every figure in the answer maps to a value in the result set.

Citations the manager can audit

The SQL is visible. The tables are named. The semantic definitions are linked. If finance disputes the answer, the trail is one tap away — and identical for every store manager.

USE CASE 2 · WRAP

Priya asked a question in plain English on her phone at 8 a.m. By 8:00:04, she had a number, a reason, a recommendation, and a citation. No analyst was woken up. No spreadsheet was sent. The warehouse stayed governed, the answer stayed grounded.

Whether the answer lives in a clause or a row, the six steps are the same: collect, prepare, embed, retrieve, generate, cite.
