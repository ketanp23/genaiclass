Why prompt evaluation matters
Writing a prompt is easy. Knowing whether it's a good prompt — before you get a bad output — is the skill. Prompt evaluation is the discipline of assessing a prompt's design quality independently of its output, so you can diagnose and improve before wasting time on iteration.
⏱ 5 min
Core principle: A well-evaluated prompt predicts output quality. A poorly designed prompt produces unpredictable results regardless of the model's capability.
Before
Evaluate the prompt
Catch structural flaws — missing context, vague roles, absent constraints — before they produce bad output.
During
Compare variants
When testing two versions of a prompt, evaluation criteria help you choose the better one systematically, not by gut feel.
After
Diagnose failures
When an output disappoints, evaluation pinpoints which dimension of the prompt failed — so the fix is targeted, not random.
The evaluation mindset shift
Without evaluation
"That response wasn't great. Let me try again with different wording and hope it's better."
With evaluation
"The role was too vague and I gave no format specification. I'll fix those two dimensions and the output quality will improve predictably."
The 5 evaluation dimensions — overview
1
Clarity
Is the task unambiguously stated? Could two readers interpret it differently?
2
Specificity
Does it include real data, named roles, and precise constraints — or vague placeholders?
3
Constraint quality
Are boundaries hard and enforceable, or soft and easily ignored?
4
Output definition
Is the expected format, structure, and length precisely described?
5
Context sufficiency
Does the model have everything it needs to respond without guessing?

The prompt evaluation checklist
Before sending any high-stakes prompt, run it through this 20-question checklist. It takes 90 seconds and catches the most common design flaws before they produce disappointing outputs.
⏱ 5 min
Dimension 1: Clarity (4 checks)
✓
Task has a verbAnalyze, Draft, Create, Evaluate, Summarize — not "help me with" or "tell me about"
✓
Scope is boundedThe task has a start and end — not open-ended "anything useful you can think of"
✓
No pronoun ambiguity"It," "they," "the situation" — replaced with specific nouns throughout
✓
One primary taskMultiple tasks in one prompt split focus — separate into distinct prompts if needed
Dimension 2: Specificity (4 checks)
✓
Role has expertise, not just title"CFO specializing in healthcare cost accounting" vs. "a CFO"
✓
At least one real numberA metric, percentage, dollar amount, or count — not "significant" or "substantial"
✓
Named entities where relevantOrg names, person names, product names — not "our company" or "the team"
✓
No placeholder language"[Insert context here]" replaced with actual context — or flagged as a known gap
Dimension 3: Constraint quality (4 checks)
✓
At least one hard exclusion"Do not recommend X" or "Exclude Y from scope" — a binary rule, not a preference
✓
Budget / word / time limits are numbers"Under 300 words" not "keep it brief." "$2M cap" not "realistic budget"
✓
Tone direction is specific"No jargon, 7th grade reading level" not "easy to understand." "Confident, not promotional" not "professional"
✓
Regulatory / legal constraints namedFor healthcare, finance, legal, HR — cite the specific rules that apply (HIPAA, union contract, SEC)
Dimension 4: Output definition (4 checks)
✓
Format is namedTable, memo, email, JSON, bulleted list, numbered list, narrative — not just "an answer"
✓
Sections or columns are definedFor tables: column names. For documents: section titles. For emails: paragraph purposes
✓
Length is specifiedWord count, page limit, or item count — prevents the model from deciding how long is "right"
✓
Audience is namedWho reads this output? Board? LP? Patient? Customer? This changes vocabulary, depth, and tone
Dimension 5: Context sufficiency (4 checks)
✓
No fact the model must inventAny statistic, name, or event the model needs but wasn't given — will be fabricated
✓
Decision alternatives are statedFor recommendations — the options being considered are explicit, not left to the model
✓
Known gaps are flaggedIf you're missing data, say "I don't have X — flag any assumption you make about it"
✓
Meta-instructions are presentAt least one of: confidence rating, assumption flagging, follow-up questions, uncertainty flags
Score interpretation: 17–20 checks passed = send with confidence. 13–16 = revise before sending high-stakes prompts. Below 13 = significant redesign needed before use in a professional context.
The one-question evaluation shortcut
When you don't have time for a full 20-check evaluation, ask yourself one question:
"If I gave this prompt to a new employee on their first day, would they produce the output I actually need?"
If the answer is no — they'd need to ask clarifying questions — then so will the AI. Every clarification the model must make internally becomes a guess. Every guess is a potential failure point. The checklist converts guesses into explicit instructions.
